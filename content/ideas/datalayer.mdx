## The Problem: The Integration Tax Is Killing Data Teams

Every data team in the world faces the same nightmare: connecting data sources to analytics tools. A typical mid-market company uses 15–30 SaaS applications, 2–3 databases, and 2–4 BI or analytics tools. Getting data from point A to point B requires building and maintaining custom ETL pipelines, and every new source or destination multiplies the complexity.

The result is the **integration tax** — the engineering hours, infrastructure costs, and opportunity costs of building plumbing instead of insights. Data engineers spend **40–60% of their time** on pipeline maintenance rather than analysis. For a company with a five-person data team, that translates to $400K–$600K annually in wasted salary alone.

**DataLayer** is a universal API that sits between data sources and BI tools, providing a standardized interface that eliminates the N-to-N integration problem. Connect once, query anywhere.

### Why Now?

1. **SaaS sprawl acceleration**: The average company's SaaS stack grew 40% between 2022 and 2025. Every new tool is another data silo.
2. **Multi-BI reality**: Companies increasingly use Looker for embedded analytics, Tableau for executive dashboards, and dbt for transformation. Each needs its own connection layer.
3. **Semantic layer maturity**: The concept of a universal metrics layer (pioneered by dbt Metrics, Cube.dev, and AtScale) has educated the market. DataLayer extends this to the full integration stack.
4. **Real-time expectations**: Batch ETL is no longer sufficient. Business users expect dashboards that update in minutes, not hours. Traditional pipelines cannot keep up.

## The Product

### Core Features

**1. Universal Connectors**
Pre-built connectors to 200+ data sources (SaaS apps, databases, data warehouses, APIs) and 50+ destinations (BI tools, operational systems, reverse ETL targets). A single connection model that works for all.

**2. Semantic Layer**
Define metrics, dimensions, and relationships once. Every downstream tool sees the same definitions. No more "revenue means different things in different dashboards." Version-controlled, with lineage tracking.

**3. Intelligent Caching**
Query results are cached at the semantic layer with configurable freshness. Most BI queries hit the cache instead of the source, reducing warehouse costs by 40–60% and improving dashboard load times by 10x.

**4. Query Federation**
Join data across sources without moving it first. Query your Salesforce CRM data alongside your PostgreSQL product data and your Stripe billing data — in a single SQL statement.

**5. Access Control**
Row-level and column-level security defined once in the semantic layer, enforced across every connected BI tool. Compliance teams configure access in one place rather than in each tool.

**6. DataLayer API**
Every metric and dataset is accessible via a REST or GraphQL API. Build internal tools, power customer-facing analytics, or feed ML models — all from the same semantic definitions.

### Technical Architecture

- **Connector Framework**: Rust-based extraction engine for high-throughput, low-latency data movement. Plugin architecture for custom connectors.
- **Semantic Engine**: Apache Arrow-based columnar processing. SQL parser that rewrites queries against the semantic model into optimized source queries.
- **Cache Layer**: Distributed cache (Redis + Apache DataFusion) with materialization policies per metric.
- **Federation Engine**: Query planner that pushes predicates to sources and performs cross-source joins locally.
- **API Gateway**: GraphQL and REST endpoints with authentication, rate limiting, and usage tracking.
- **Infrastructure**: Kubernetes-native. Available as SaaS or self-hosted (VPC deployment for enterprise).
